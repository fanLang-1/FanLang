# 凡语 · 场景 · 符号映射表

> 常见任务 × 推荐组织形态 × 凡语符号组合。AI 读取后按此执行。

---

## 【总】设计原则

凡语的 25 个符号不是给用户学的，是给 AI 调度使用的。本文件定义"遇到某类任务时，应当启动哪种组织形态、调用哪些符号角色"。

---

## 【定】场景表

### 写文章 / 报告

| 属性 | 值 |
|:--|:--|
| 触发词 | "写一篇""帮我写""写份报告""生成文章" |
| 推荐组织 | Pipeline（王写手→李编辑→张校对） |
| 符号序列 | 写 → 改 → 查 |
| 说明 | 串行流水线保证文字质量、事实准确 |
| 示例 | `quick_pipeline(hall).execute("写一篇AI发展分析")` |

### 翻译

| 属性 | 值 |
|:--|:--|
| 触发词 | "翻译""翻一下""translate" |
| 推荐组织 | Pipeline（王写手→李编辑） |
| 符号序列 | 翻 → 改 |
| 说明 | 初译+润色双环，2个角色即可 |

| 属性 | 值 |
|:--|:--|

| 符号序列 | 问 |
| 说明 | 简单问题不用组织，不要过度设计 |
| 示例 | `hall.discuss(hall.get_role("王写手"), "什么是RAG")` |

## 【理】符号→组织形态自动路由

```
输入检测 → 符号识别

  写/翻/编  → Pipeline（有工序的任务）
  想/画      → Panel（需要多样性的任务）
  评/比      → Adversarial（需要裁决的任务）
  问/查/解   → 单次调用（信息查询）
  理/总/记   → Pipeline（输入加工→输出）
  阅/兼/定   → 待实现（框架符号需要OODA元控制）
```

## 【解】符号能力矩阵

| 符号 | 流水线角色 | 评议角色 | 对撞角色 |
|:--|:--|:--|:--|
| ✍️ 写 | 产出者 | 版本A产出者 | 正方（立论） |
| 🔄 改 | 润色者 | - | - |
| 🔍 查 | 校验者 | 数据提供者 | - |
| 💡 想 | - | 版本B产出者 | - |
| 🗂️ 理 | 结构化 | 数据整理者 | - |
| ⭐ 评 | - | 主编（汇总） | 裁判（裁决） |
| 🔑 解 | - | - | 反方（驳斥） |
| ⏰ 定 | 调度者 | 调度者 | 调度者 |
