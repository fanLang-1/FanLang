# 🏗️ 中文 AI 语言 · 架构设计

## 选型：OODA 层次模型

在 OSI 7层、Clean Architecture、OODA 等候选框架中，
**最合适的是 OODA 循环 + 5层架构**。

为什么：
- OODA 本身就是一个循环（Loop），不需要额外引入
- 用户输入→澄清→执行→反馈→迭代，就是 OODA 的天然映射
- 普通人也听得懂"观察→理解→决定→行动"
- 军事领域用了60年，证明了它在"不确定性下快速决策"的有效性

## 5 层架构（OODA 层次模型）

```
                  用户（大众）
                      │
┌──────────────────────▼──────────────────────┐
│  Layer 5: 用户界面层 (Interface)            │
│  12个大按钮 + 输入框 + 结果展示              │
│  【写改翻总问画想解教比查...】               │
└──────────────────────┬──────────────────────┘
                       │  用户输入
┌──────────────────────▼──────────────────────┐
│  Layer 4: 理解层 (Orient)           ← 关键  │
│  追问澄清用户需求：                           │
│  "你想写什么类型？多长？什么风格？"           │
│  输出：结构化的需求说明书                     │
└──────────────────────┬──────────────────────┘
                       │  结构化需求
┌──────────────────────▼──────────────────────┐
│  Layer 3: 调度层 (Decide)                    │
│  选智能体、选工具、选模型                      │
│  接入开源智能体：AutoGen / CrewAI / LangGraph │
│  输出：任务计划                               │
└──────────────────────┬──────────────────────┘
                       │  任务指令
┌──────────────────────▼──────────────────────┐
│  Layer 2: 执行层 (Act)                       │
│  调用 Ollama / API / 知识库                    │
│  多步骤执行、工具调用                          │
│  输出：原始结果                               │
└──────────────────────┬──────────────────────┘
                       │  结果
┌──────────────────────▼──────────────────────┐
│  Layer 1: 反馈层 (Observe)                    │
│  检查结果质量 → 不满意？→ 重新进入OODA循环   │
│  用户反馈 → 调整 → 重新执行                   │
└──────────────────────┬──────────────────────┘
                       │  用户确认
                       ▼
                   完成/输出
```

## 核心流程（带澄清的 OODA）

用户输入一个模糊需求：

```
用户: "帮我写点东西"
                  │
     Layer 4: Orient ──→ "写什么类型？文章/文案/代码/报告？"
     ← 用户: "文章"
     Layer 4: Orient ──→ "什么主题？多长？什么风格？"
     ← 用户: "AI入门，500字，口语化"
                  │
     Layer 3: Decide ──→ 选【写】符号，选 writing_agent
                  │
     Layer 2: Act ─────→ 执行生成
                  │
     Layer 1: Observe ─→ "结果满意吗？要改吗？"
     ← 用户: "再口语化一点"
                  │
     ──────→ 回到 Orient，进入下一轮 OODA ←─────
```

## 开源智能体接入点

```
Layer 3 ── 调度层
     │
     ├── AutoGen (微软)     → 多智能体对话协作
     ├── CrewAI             → 角色扮演式智能体团队
     ├── LangGraph (LangChain) → 图式工作流
     └── (未来) 用符号定义智能体 → 中文原生的智能体语言
```

每个智能体配一个中文名：
- writing_agent → 【写】智能体
- translating_agent → 【翻】智能体  
- searching_agent → 【查】智能体
- ...

## 与符号体系的映射

```
25个符号 = 25种智能体类型
OODA循环 = 系统默认行为（用户不需要知道）
Layer 4 的追问 = 系统自动补充缺失信息
```

## 产品形态方向

```
第一阶段: Web 界面（现在的脚手架）← 你现在的位置
第二阶段: 桌面 App（Tauri / Electron）
第三阶段: 手机小程序 / App
第四阶段: 嵌入其他产品的 AI 能力层
```

---

**一句话总结**：
OODA层次模型 = 分层清晰（看得懂） + 循环内建（不用额外加） + 澄清优先（减少AI幻觉） + 可接智能体（不闭门造车）
